## Visão geral

Criar uma base de conhecimento não é suficiente para que o agente a use — é preciso conectar explicitamente a base ao agente. Essa conexão é feita na aba de treinamento do agente e pode ser configurada com parâmetros de busca que controlam quantos e quão relevantes precisam ser os resultados retornados.

Um agente pode ter múltiplas bases conectadas simultaneamente. Nesse caso, a busca é realizada em todas as bases e os resultados são mesclados antes de serem injetados no contexto do LLM.

## Pré-requisitos

- Uma [base de conhecimento criada](/guides/knowledge/creating) com ao menos um documento no status **Indexado**.
- Role `manager`, `trainer` ou `owner` no workspace.
- Um agente criado no workspace.

## Como conectar

<Steps>
  <Step title="Abra o agente">
    No menu lateral, acesse **Agentes** e clique no agente que receberá a base de conhecimento.
  </Step>
  <Step title="Acesse a aba Treinamento">
    Dentro da tela do agente, clique na aba **Treinamento** (ou **Knowledge**, dependendo da versão da interface). Esta aba centraliza todas as fontes de conhecimento do agente.
  </Step>
  <Step title="Adicione a base de conhecimento">
    Na seção **Bases de conhecimento**, clique em **+ Adicionar base**. Uma lista das bases disponíveis no workspace será exibida. Selecione a base desejada e confirme.
  </Step>
  <Step title="Configure os parâmetros de retrieval">
    Após adicionar a base, os parâmetros de busca ficam disponíveis para edição:

    - **Top-k** — número máximo de chunks retornados por consulta. Padrão: `5`. Valores maiores trazem mais contexto, mas aumentam o tamanho do prompt enviado ao LLM.
    - **Threshold** — limiar mínimo de similaridade coseno (0 a 1). Padrão: `0.7`. Chunks com similaridade abaixo desse valor são descartados. Valores mais baixos retornam mais resultados (mais abrangente); valores mais altos exigem maior proximidade semântica (mais preciso).
  </Step>
  <Step title="Salve a configuração">
    Clique em **Salvar**. A base passa a ser consultada automaticamente em toda conversa atendida por esse agente a partir desse momento.
  </Step>
</Steps>

## Parâmetros de retrieval em detalhe

### Top-k

O `top-k` define quantos chunks (fragmentos de texto) são recuperados da base e injetados no contexto do LLM antes de cada resposta do agente.

| Top-k | Efeito |
|-------|--------|
| 3 | Respostas mais focadas, menor custo de tokens, risco de perder contexto relevante |
| 5 | Equilíbrio padrão entre abrangência e custo |
| 10 | Mais contexto disponível para perguntas complexas, mas aumenta o tamanho do prompt |

<Tip>
  Para bases de Q&A com respostas curtas e objetivas, um top-k de 3 é suficiente. Para bases com documentos técnicos longos onde o contexto pode estar distribuído em múltiplos trechos, use top-k de 7 a 10.
</Tip>

### Threshold de similaridade

O `threshold` é o critério de qualidade mínima para que um chunk seja incluído na resposta. É um valor de similaridade coseno entre 0 e 1:

- **0.9** — apenas resultados muito próximos semanticamente. Poucos falsos positivos, mas pode retornar zero resultados se o conteúdo não for exato.
- **0.7** — equilíbrio padrão recomendado para a maioria das bases.
- **0.5** — mais abrangente; inclui resultados vagamente relacionados. Útil para bases com linguagem variada onde a exatidão semântica é menor.

Se o agente "não sabe" algo que deveria saber, o primeiro ajuste é reduzir o threshold. Se o agente traz informações tangencialmente relacionadas que confundem a resposta, aumente o threshold.

## Múltiplas bases por agente

Um agente pode ter várias bases conectadas. O comportamento nesse caso:

- A busca é executada em **todas as bases simultaneamente**.
- Os resultados de cada base são ordenados por similaridade.
- Os `top-k` chunks mais relevantes do conjunto total são selecionados (não top-k por base).
- Não há diferenciação de fonte nos chunks injetados no contexto — o LLM recebe o texto, não o nome da base.

### Prioridade entre bases

Na aba de treinamento, você pode reordenar as bases conectadas. A ordem define a prioridade quando dois chunks de bases diferentes têm similaridade idêntica — o chunk da base com maior prioridade é preferido no desempate.

Para separar conhecimentos com propósitos muito distintos (ex.: FAQ de suporte vs. catálogo de produtos), prefira conectar bases separadas ao invés de misturar tudo numa base única. Isso facilita a manutenção e permite desconectar uma base sem afetar a outra.

## Desconectando uma base

Para remover uma base de um agente sem excluir a base:

1. Acesse a aba **Treinamento** do agente.
2. Encontre a base na lista.
3. Clique no ícone de remoção (lixeira ou X) ao lado da base.
4. Confirme a remoção.

A base continua existindo no workspace e pode ser reconectada ao mesmo agente ou a outros agentes a qualquer momento. Nenhum dado de indexação é perdido.

## Testando a conexão

Após conectar a base, use o chat de teste do agente para verificar se o retrieval está funcionando:

1. Abra o agente e acesse a aba **Testar** (ou o chat de preview).
2. Faça uma pergunta diretamente relacionada a um conteúdo indexado na base.
3. Se a resposta for precisa e baseada no conteúdo da base, o retrieval está funcionando.
4. Se o agente responder de forma genérica ou errada, verifique o status dos documentos (devem estar **Indexados**) e tente reduzir o threshold.

## Boas práticas

- Sempre verifique o status de processamento dos documentos antes de fazer o primeiro teste. Documentos em status **Processando** ainda não contribuem para as buscas.
- Ajuste o threshold de forma incremental — reduza em 0.05 por vez e reavalie após cada ajuste em vez de fazer mudanças drásticas.
- Para bases com conteúdo muito específico (ex.: tabela de preços com números exatos), prefira adicionar esse conteúdo como Q&A em vez de documento — Q&As são indexados com mais controle sobre a granularidade.
- Monitore os misses de retrieval usando o [painel de analytics](/guides/analytics) para identificar perguntas frequentes que a base não consegue responder.
